## Fix #29 — Dev API fetch bypasses centralized client

**File:** `desk/src/pages/DocTypeBuilder.tsx` (line 266)

**Problem:** The `useEffect` that fetches available apps uses raw `fetch("/api/v1/dev/apps")`, bypassing the centralized `request()` function in `desk/src/api/client.ts`. This means no `Authorization` header and no `X-Moca-Site` header are sent, causing the backend to return 400 `TENANT_REQUIRED`. The App/Module dropdowns stay empty.

**Fix:**
1. Add `get` to the existing import from `@/api/client` (line 5 already imports `post, put`).
2. Replace the raw `fetch()` call with `get<ApiResponse<...>>("dev/apps")`.
3. Use the centralized error handling (the `get()` helper throws `MocaApiError` on failure).

**Before:**
```typescript
useEffect(() => {
  fetch("/api/v1/dev/apps")
    .then((r) => (r.ok ? r.json() : null))
    .then((json) => { if (json?.data) setAppList(json.data); })
    .catch(() => {});
}, []);
```

**After:**
```typescript
useEffect(() => {
  get<ApiResponse<{ name: string; modules: string[] }[]>>("dev/apps")
    .then((res) => setAppList(res.data))
    .catch(() => {});
}, []);
```

**Files changed:** 1 (`DocTypeBuilder.tsx`)
**Lines changed:** ~5

## Fix #30 — No way to delete a column

**Files:** `desk/src/components/doctype-builder/SectionNode.tsx`, `desk/src/components/doctype-builder/ColumnNode.tsx`

**Problem:** Once a column is added via "+ Add Column", there is no UI to remove it. The store already implements `removeColumn(tabIdx, sectionIdx, columnIdx)` with a guard that keeps at least one column per section. The method is just not wired to any UI component.

**Fix:**

1. **SectionNode.tsx** — Pass `onRemoveColumn` callback and `totalColumns` count to each `ColumnNode`.

2. **ColumnNode.tsx** — Add a `Trash2` icon button at the top-right corner of the column container, visible on hover (using the existing `group`/`opacity-0 group-hover:opacity-100` pattern from tab options). Only render when `totalColumns > 1`.

**Design rationale:** A simple hover-reveal trash button rather than a dropdown menu, because columns don't need rename or other options (unlike tabs and sections). This keeps the interaction lightweight and consistent with the minimal column UI.

**Props added to ColumnNode:**
```typescript
interface ColumnNodeProps {
  column: ColumnDef;
  tabIdx: number;
  sectionIdx: number;
  colIdx: number;
  onRemoveColumn: () => void;  // NEW
  totalColumns: number;         // NEW
}
```

**Files changed:** 2 (`SectionNode.tsx`, `ColumnNode.tsx`)
**Lines changed:** ~20

## Fix #31 — DocType name validation not surfaced

**File:** `desk/src/components/builder-kit/BuilderToolbar.tsx`

**Problem:** The name input accepts any text. The backend requires PascalCase with only letters and digits. Users naturally type "Library Member" or "library-member", get a confusing toast error on save, and have no guidance about the expected format.

**Fix:** Auto-format on input. Add a `toPascalCase` helper that transforms input as the user types:
- Strip all non-alphanumeric characters except spaces, hyphens, and underscores (word boundary markers)
- Split on word boundaries (space, hyphen, underscore, or camelCase boundary)
- Capitalize the first letter of each segment, lowercase the rest
- Join without separators

**Examples:**
| Input | Output |
|-------|--------|
| `library member` | `LibraryMember` |
| `my-thing` | `MyThing` |
| `hello_world` | `HelloWorld` |
| `libraryMember` | `LibraryMember` |
| `ABC` | `Abc` |
| `hello world 123` | `HelloWorld123` |

**Additional changes:**
- Update placeholder from `"Untitled"` to `"e.g. LibraryMember"`
- The `onNameChange` prop receives the already-transformed value

**Files changed:** 1 (`BuilderToolbar.tsx`)
**Lines changed:** ~15

## Fix #32 — field_type empty string on save

**Files:** `desk/src/pages/DocTypeBuilder.tsx`, `pkg/api/dev_handler.go`

**Problem:** When a DocType is saved, the `field_type` property in the persisted JSON is an empty string. The field type is correctly shown in the UI (badge on canvas, value in property panel combobox), so it exists in the store. The issue occurs during serialization or transmission.

**Investigation notes:** The store's `addField()` correctly sets `field_type: fieldType`. The `handleSave` passes `store.fields` directly. The Go backend's `buildDocTypeJSON` iterates the fields map and marshals each `FieldDef`. The exact trigger for the empty value could not be pinpointed statically — it may involve a re-render cycle in the PropertyPanel's Radix Select component or a field-rename key mismatch.

**Fix (two-layer defense):**

1. **Frontend validation (DocTypeBuilder.tsx `handleSave`):** Before sending the payload, check that every field in `store.fields` has a non-empty `field_type`. If any is empty, show a toast error identifying the field and block the save. This prevents bad data from reaching the backend.

2. **Backend validation (dev_handler.go):** In `HandleCreateDocType` and `HandleUpdateDocType`, after decoding the request body, validate that each field in `req.Fields` has a non-empty `FieldType`. Return 400 with a clear message like `"field 'full_name' has no field_type"`.

**Files changed:** 2 (`DocTypeBuilder.tsx`, `dev_handler.go`)
**Lines changed:** ~10
