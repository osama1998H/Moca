# v1.0.4 DX Issue Fixes — Design

**Date:** 2026-04-16
**Scope source:** `docs/dx/dx-test-session-v1.0.4-report.md`
**Delivery:** single PR, titled `fix(dx): resolve v1.0.4 DX blockers and follow-up issues (#33-#38)`

## Goals

- Unblock the Desk browser login flow after server restarts (#35).
- Make the DocType Builder functional end-to-end — creates files, registers in DB, exposes REST endpoints (#36, #37).
- Cut first-project friction: no manual `desk/.env` edit (#34), actionable failure messages + retry-safe state on `moca init` (#33).
- Quiet benign WARN spam in single-instance mode (#38).

## Non-goals

- Root-cause investigation of the intermittent desk scaffolding failure itself. Issue #33 makes the next occurrence observable; a captured failure log is the prerequisite for the real RCA.
- Re-architecting the dev handler's context-key layering (beyond the one-line fix in #36).
- Adding an API settings UI to the builder. Issue #37 is server-side default injection only; a builder UI change is a later enhancement.
- Touching `@osama1998h/desk` npm package internals. Everything in this spec stays in the `moca` CLI and the project-scaffolded `desk/`.

## Issue #35 (Blocker) — Stale JWT blocks login

### Root cause chain

1. `desk/src/api/client.ts:26` — module-level `accessToken` is seeded from `localStorage` at script load.
2. `desk/src/providers/AuthProvider.tsx:81-84` — `restore()` returns early when no refresh token exists, without clearing the seeded `accessToken`.
3. `client.ts:113-115` — every request attaches `Authorization: Bearer <stale>`, including `POST /auth/login`.
4. `pkg/api/middleware.go:222-226` — auth skip list is `{"/desk/", "/health", "/ws"}`; `/api/v1/auth/*` is not skipped.
5. `pkg/auth/authenticator.go:60-67` — an invalid Bearer token returns `err`, not Guest fallback.
6. `middleware.go:257-260` — that error becomes `401 AUTH_FAILED` before the login handler ever runs.

### Backend fix — `pkg/api/middleware.go`

Add a version-agnostic `isPublicAuthPath(path string) bool` helper next to `authMiddleware`. It matches any `/api/*/auth/<tail>` where `<tail>` is in the public set:

| Route | Skip auth? | Why |
|-------|------------|-----|
| `POST /api/{v}/auth/login` | yes | mints tokens from credentials |
| `POST /api/{v}/auth/refresh` | yes | mints tokens from refresh token |
| `POST /api/{v}/auth/logout` | **no** | needs current bearer to revoke session; frontend swallows 401 |
| `GET  /api/{v}/auth/sso/authorize` | yes | initiates SSO; no user yet |
| `GET  /api/{v}/auth/sso/callback` | yes | IdP redirect; mints session |
| `GET  /api/{v}/auth/saml/metadata` | yes | public SP metadata |
| `POST /api/{v}/auth/saml/acs` | yes | SAML assertion consumer; mints session |

Extend the skip branch at `middleware.go:222-226` to call `isPublicAuthPath`.

### Frontend fix — `desk/src/providers/AuthProvider.tsx:80-84`

```ts
async function restore() {
  const rt = getRefreshToken();
  if (!rt) {
    setAccessToken(null); // clear orphaned access token
    setIsLoading(false);
    return;
  }
  // …unchanged
}
```

### Tests

- `pkg/api/middleware_test.go`: extend auth middleware tests with one case per exempt path (skip auth, pass through) and one for logout (still requires auth). Negative test: `POST /api/v1/auth/login` with a garbage Bearer header → 200.
- `desk/src/providers/AuthProvider.test.tsx` (new): mount `AuthProvider` with `localStorage` seeded only with `moca_access_token` (no refresh) → assert `localStorage.getItem("moca_access_token") === null` after mount.

## Issue #36 (Blocker) — Dev API uses wrong site

### Root cause

`pkg/api/dev_handler.go:449-454`:

```go
func siteFromContext(r *http.Request) string {
    if site, ok := r.Context().Value("site").(string); ok { return site }
    return "default"
}
```

Tenant middleware (`middleware.go:160` → `context.go:40` `WithSite`) stores `*tenancy.SiteContext` under typed key `siteKey`. The dev handler reads raw string key `"site"` — they never match, so `siteFromContext` always returns `"default"`. The registry then tries to register in schema `default`, which has no `tab_doctype` table.

### Fix

Delete `siteFromContext` entirely. Replace both call sites (`dev_handler.go:241` in `HandleCreateDocType` and `dev_handler.go:314` in `HandleUpdateDocType`) with:

```go
site := SiteFromContext(r.Context())
if site == nil {
    writeJSON(w, http.StatusBadRequest, map[string]string{"error": "site context required"})
    return
}
// …
h.registry.Register(r.Context(), site.Name, data)
```

The `nil`-guard is defensive against test harnesses that bypass `tenantMiddleware`. Package is `api` so the call is unqualified.

### Ripple check

`siteFromContext` has exactly two call sites (grepped). Removing it is safe.

### Tests

- `pkg/api/dev_handler_test.go`: extend `TestHandleCreateDocType_*` with a fake `tenantMiddleware` that injects `*tenancy.SiteContext{Name: "acme"}` via `WithSite`. Use a test-double `*meta.Registry` that records the `site` argument passed to `Register`. Assert it equals `"acme"`.
- New test: no site in context → 400 `"site context required"`.
- `pkg/api/dev_api_integration_test.go`: extend with a site-context scenario.

## Issue #37 (Major) — DocType Builder omits `api_config`

### Root cause

`pkg/api/dev_handler.go:415-438` — `buildDocTypeJSON` does not emit `api_config`. The request struct `DevDocTypeRequest` (lines 137-146) has no `APIConfig` field. Builder-created DocTypes therefore have no `api_config` block, so the auto-generated REST layer skips them (routes register only when `APIConfig.Enabled == true`).

### Target shape

`pkg/meta/stubs.go:129-150` confirms `APIConfig` has `Enabled`, `AllowList`, `AllowGet`, `AllowCreate`, `AllowUpdate`, `AllowDelete`, `AllowCount`, `AllowBulk`, `DefaultPageSize`, `MaxPageSize`. The scaffolded template `internal/scaffold/templates.go:146-156` already uses these. Port the same shape to `buildDocTypeJSON` for byte-for-byte consistency with `moca app new --doctype`.

### Fix — backend injection

In `buildDocTypeJSON`, append:

```go
"api_config": map[string]any{
    "enabled":           true,
    "allow_list":        true,
    "allow_get":         true,
    "allow_create":      true,
    "allow_update":      true,
    "allow_delete":      true,
    "allow_count":       true,
    "default_page_size": 20,
    "max_page_size":     100,
},
```

An explicit map (rather than marshaling `meta.APIConfig{}`) keeps slice fields absent rather than serialized as `null`, matching the scaffold template.

### Scope

The default applies unconditionally on create since `DevDocTypeRequest` has no `APIConfig` field. A future builder change will extend the request with `APIConfig *meta.APIConfig`, at which point the default becomes a fallback when the field is nil.

### Tests

- `pkg/api/dev_handler_test.go`: new `TestHandleCreateDocType_EmitsDefaultAPIConfig` — POST a minimal create request, read the written file, assert `api_config.enabled == true` and allow-* flags.
- Integration: register a created DocType through `meta.Registry` and confirm REST routes are generated — this is the end-to-end behavior assertion the issue actually blocks.

## Issue #34 (Major) — Empty `VITE_MOCA_SITE` in scaffolded `desk/.env`

### Approach

Update `desk/.env` from the CLI on `moca site use` and on `moca site create` (first-site bootstrap case), preserving existing file content.

### Where to write

- `runSiteUse` in `cmd/moca/site.go:361-389` — after successfully writing `.moca/current_site`, update `desk/.env`.
- `runSiteCreate` — when `.moca/current_site` doesn't exist yet (first site), call the same helper immediately after `SetActiveSite`.

Skipped: `site drop`, `site rename`. Rare operations, and the helper call can be added there in a follow-up if the need arises.

### Helper — new `internal/scaffold/deskenv.go`

```go
// UpdateDeskEnvSite sets VITE_MOCA_SITE in <projectRoot>/desk/.env.
// Returns nil if desk/ is absent (headless project).
// Preserves existing non-VITE_MOCA_SITE lines and comments.
// Does not overwrite a non-empty VITE_MOCA_SITE unless force is true.
func UpdateDeskEnvSite(projectRoot, siteName string, force bool) error
```

Semantics:

1. If `<projectRoot>/desk` does not exist → return nil.
2. If `<projectRoot>/desk/.env` does not exist → create it with `VITE_MOCA_SITE=<siteName>\n`.
3. Else read file, split into lines, find `VITE_MOCA_SITE=`:
   - Present and empty → replace with `siteName`.
   - Present and non-empty, `force=false` → leave alone.
   - Present and non-empty, `force=true` → replace.
   - Absent → append `VITE_MOCA_SITE=<siteName>\n`.
4. Write atomically via `os.CreateTemp` + `os.Rename` in the same directory.

`site use` and `site create` call with `force=false`.

### Call sites

- `runSiteUse`: after `.moca/current_site` write, call helper. Warn via `w.PrintWarning(…)` on error; do not fail.
- `runSiteCreate`: first-site branch, same call, same warn-on-error handling.

### Tests

`internal/scaffold/deskenv_test.go` — table-driven:

- no `desk/` → no-op, no error
- empty `.env` → create with single `VITE_MOCA_SITE=foo`
- `VITE_MOCA_SITE=` → update to `foo`
- `VITE_MOCA_SITE=existing` + `force=false` → unchanged
- `VITE_MOCA_SITE=existing` + `force=true` → replaced
- key absent → append
- `VITE_OTHER=x` alongside → `VITE_OTHER` preserved
- atomic rename leaves no partial file on simulated write failure

`cmd/moca/integration_test.go` already covers `site use` around line 395; extend to assert `desk/.env` contents.

## Issue #33 (Major) — `moca init` desk scaffolding failure is opaque

This PR makes the next failure observable and retry-safe. It does not fix the intermittent failure itself; that needs a captured failure log.

### Problem A — wrapped error is hidden in rich format

`CLIError.Format` in `internal/output/error.go:39-65` prints `Message`, `Context`, `Cause`, `Fix`, `Reference` but never the wrapped `Err`. `ScaffoldDesk` returns errors wrapped with `%w`, preserved via `.WithErr(err)` in `init.go:120`, but `Format()` silently discards them. Users see only `"Failed to scaffold desk frontend"` with no detail.

### Fix A — surface wrapped err when `Cause` is unset

In `CLIError.Format`, before the existing `Cause` print block:

```go
cause := e.Cause
if cause == "" && e.Err != nil {
    cause = e.Err.Error()
}
if cause != "" { /* existing block, using `cause` */ }
```

Explicit `.WithCause(…)` still wins. This is additive and fixes every `.WithErr(err)` caller in the CLI — all currently losing detail in rich format.

### Problem B — half-written `desk/` prevents retry

On scaffold failure, `deskDir` and any subdirectories remain. `init.go:68-72` checks `moca.yaml` existence but not a stale `desk/`. Next `moca init` to the same path fails at `desk.go:39-41` with `"desk/ directory already exists"`. Users currently retry to a different path.

### Fix B — clean up `desk/` on scaffold failure

In `cmd/moca/init.go:113-122`:

```go
if err := scaffold.ScaffoldDesk(...); err != nil {
    if rmErr := os.RemoveAll(filepath.Join(targetDir, "desk")); rmErr != nil {
        err = fmt.Errorf("%w (cleanup also failed: %v)", err, rmErr)
    }
    s.Stop("Failed")
    return output.NewCLIError("Failed to scaffold desk frontend").WithErr(err)
}
```

Cleanup is strictly scoped to `filepath.Join(targetDir, "desk")`. A pre-existing `desk/` short-circuits `ScaffoldDesk` at line 40 before any state is created, so the cleanup branch only runs when the partial state was created by this invocation.

### Tests

- `internal/output/error_test.go`: extend `TestCLIError_Format`:
  - `WithErr(errors.New("boom"))` alone → output contains `"Cause:\n  boom"`.
  - `WithErr(...)` + `WithCause("explicit")` → output contains `"Cause:\n  explicit"`.
  - `WithErr(nil)` + `WithCause("")` → no Cause block.
- `cmd/moca/init_test.go` (or focused unit test on a refactored `scaffoldDeskWithCleanup` helper): simulate scaffold failure, assert `desk/` is absent after the command returns.

## Issue #38 (Minor) — Leader election log spam

### Fix

`pkg/queue/leader.go:232` — downgrade to `Debug` and add context:

```go
le.logger.Debug("leader election: lock lost, will re-acquire",
    slog.String("key", le.config.Key),
    slog.String("instance", le.config.InstanceID),
)
```

### Rationale

- In multi-instance deployments, lock-lost events are expected during failover. Alerting should key off "no leader for N minutes," not this log line.
- In single-instance mode, the event is noise — likely triggered by a transient Redis connection blip or clock skew once every ~15 min.
- No dashboards, tests, or alerts key off this string (grepped).
- `Debug` keeps the information available for investigation without polluting default output.

### Not doing

- Dynamic single-instance detection: no clean signal; adds state for a log cleanup.
- Investigating why the lock is lost every 15 min: requires longer Redis client instrumentation; not warranted here.

### Tests

Existing `TestLeaderElection_HeartbeatRenews` and `TestLeaderElection_GracefulRelease` cover the underlying behavior. The level change is test-safe — no tests assert the WARN line.

### Risks and mitigations

- **Auth skip-list drift**: a future public auth route won't be exempt unless added to `isPublicAuthPath`. Mitigation: doc comment on the helper listing the source of truth (`AuthHandler.RegisterRoutes` + `SSOHandler.RegisterRoutes`).
- **`desk/.env` atomic write during `vite dev`**: `os.Rename` on the same filesystem is atomic on POSIX; Vite's watcher handles the swap cleanly. macOS `FSEvents` may emit two events; harmless.
- **Default `api_config` injection for existing builder DocTypes**: applies only on new create requests. Existing on-disk JSON without `api_config` is unchanged until the next save.
