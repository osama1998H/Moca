## Fix Design

### Issue #23: Wire Dev API Routes (Blocker)

**Root cause:** `DevHandler.RegisterDevRoutes()` in `pkg/api/dev_handler.go:32-39` is fully implemented with all 5 endpoints but never called during server startup in `internal/serve/server.go`.

**Files to modify:**

1. **`internal/config/types.go`** (line 120-128) — Add `DeveloperMode` field to `DevelopmentConfig`:
   ```go
   DeveloperMode bool `yaml:"developer_mode,omitempty"`
   ```
   Matches the `developer_mode: true` YAML key users set in `moca.yaml`.

2. **`internal/serve/server.go`** — Two changes:
   - Add `AppsDir string` field to `ServerConfig` struct (line 38-45)
   - After the pprof block (~line 350), conditionally instantiate and register the dev handler:
     ```go
     if cfg.Config != nil && cfg.Config.Development.DeveloperMode && cfg.AppsDir != "" {
         devHandler := api.NewDevHandler(cfg.AppsDir, registry, logger)
         devHandler.RegisterDevRoutes(gw.Mux(), "v1")
         logger.Info("dev API endpoints enabled at /api/v1/dev/")
     }
     ```

3. **`cmd/moca/serve.go`** (line 76-83) — Pass `AppsDir` in `ServerConfig`:
   ```go
   AppsDir: filepath.Join(projectRoot, "apps"),
   ```
   This path is already computed at line 135 for the meta watcher.

4. **`cmd/moca-server/main.go`** (line 53-58) — Pass `AppsDir` in `ServerConfig`. This binary runs from the project root (CWD), so use `"apps"` (relative path) or resolve via `filepath.Abs("apps")`.

### Issue #27: `moca init` should set `developer_mode: true`

**Root cause:** `cmd/moca/init.go:272-276` builds initial `DevelopmentConfig` without the `DeveloperMode` field.

**File to modify:**

- **`cmd/moca/init.go`** (line 272-276) — Add `DeveloperMode: true` to the `Development` config:
  ```go
  Development: config.DevelopmentConfig{
      Port:          8000,
      Workers:       1,
      AutoReload:    true,
      DeveloperMode: true,
  },
  ```

Safe because `moca init` only creates development projects. Production deployments override this.

### Issue #24: App/Module Selectors with Dropdown

**Root cause:** `DocTypeBuilder.tsx:329-350` uses plain `<Input>` elements with no API integration.

**Files to modify:**

- **`desk/src/pages/DocTypeBuilder.tsx`** (lines 329-350) — Replace `<Input>` elements with `<select>` dropdowns:
  - App `<select>`: Populated from `GET /api/v1/dev/apps` response
  - Module `<select>`: Populated from the selected app's `modules` array (no separate API call)
  - When App changes, auto-clear Module
  - On mount, fetch apps list once via `useEffect`

**Data flow:**
1. Component mounts -> `fetch("/api/v1/dev/apps")` -> store app list in local state
2. App `<select>` shows app names from fetched list
3. When app selected -> filter modules from that app's data -> populate Module `<select>`
4. Both feed into existing `store.setApp()` / `store.setModule()`
5. If dev API unavailable, selectors show empty options (graceful fallback)

### Issue #22: Tab Rename via Double-Click

**Root cause:** `SchematicCanvas.tsx:39-43` renders tab labels as plain text inside `TabsTrigger` with no edit affordance.

**File to modify:**

- **`desk/src/components/doctype-builder/SchematicCanvas.tsx`** (lines 39-43):
  - Add `editingTab: number | null` local state
  - On **double-click** of a TabsTrigger, set `editingTab = idx` and render an `<input>` replacing the label text
  - On **blur** or **Enter**, call `store.updateTab(idx, { label: value })` and clear `editingTab`
  - On **Escape**, cancel edit and clear `editingTab`

Store already has `updateTab(tabIdx, patch)` at `doctype-builder-store.ts:286-293` — no store changes needed.

Follows the same pattern as `SectionNode.tsx:39-44` where section labels are inline-editable.

### Issue #25: Tab Delete Button

**Root cause:** `SchematicCanvas.tsx:46-54` only has a "+ Tab" button. Store has `removeTab(tabIdx)` at line 278-284 with safety check (keeps >= 1 tab) but no UI calls it.

**File to modify:**

- **`desk/src/components/doctype-builder/SchematicCanvas.tsx`** — Add a dropdown menu per tab:

  - Reuses the exact `DropdownMenu` / `DropdownMenuItem` pattern from `SectionNode.tsx:47-63`

### Issue #26: Auto-Sync Field Name from Label

**Root cause:** `doctype-builder-store.ts:234-242` `updateField()` is a simple merge with no special handling for label changes.

**Files to modify:**

1. **`desk/src/components/doctype-builder/types.ts`** — Add exported helper:
   ```ts
   export function labelToFieldName(label: string): string {
     return label.trim().toLowerCase().replace(/\s+/g, "_").replace(/[^a-z0-9_]/g, "");
   }
   ```

2. **`desk/src/pages/DocTypeBuilder.tsx`** (~line 291) — Intercept `onChange` callback:
   - When `key === "label"`, check if the current field name matches `labelToFieldName(currentLabel)`
   - If they match (name was auto-generated, never manually edited), also update `name` to `labelToFieldName(newLabel)`
   - If they don't match (user manually set the name), leave it alone
   - No extra flag needed — comparison is the detection mechanism
